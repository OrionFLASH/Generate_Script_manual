# Скрипт обновления параметров СУП (`SUP_Config_Update.js`)

Документ состоит из двух частей:

| Часть | Для кого | Содержание |
|-------|----------|------------|
| **I** | Обычный пользователь СУП | Пошаговые инструкции: что нажать, что будет, если пропустить шаг |
| **II** | Разработчик / продвинутый пользователь | API, форматы JSON, справочник кода |

Файл скрипта: `Script/SUP_Config_Update.js`. Копия для пересылки по почте: `./post_txt_sync.sh` → `POST/SUP_Config_Update.js.txt`.

---

# Часть I. Руководство для пользователя

## I.1. Что делает этот скрипт простыми словами

Скрипт — это **вспомогательное окно** в браузере, которое умеет:

1. **Скачать** текущие значения параметра с сервера (как выгрузка из СУП).
2. **Показать** их в удобном виде для правки.
3. **Сохранить обратно** на сервер — как если бы вы открыли параметр в СУП, изменили список и нажали **«Сохранить»**.

Программировать не нужно. Достаточно уметь открыть консоль браузера (один раз вставить текст файла) и нажимать кнопки на панели.

### Что скрипт умеет (полный список возможностей)

| № | Возможность | Когда использовать |
|---|-------------|-------------------|
| 1 | Скачать параметр с сервера по **имени** | Нужно узнать или изменить текущие значения |
| 2 | Сохранить скачанное в файл `.json` на компьютер | Бэкап перед правками |
| 3 | Править значения в окне и отправить на сервер | Основной сценарий |
| 4 | Загрузить готовый `.json` с компьютера | Уже есть файл выгрузки |
| 5 | Отправить **несколько** параметров из одного файла | Массовое обновление |
| 6 | Пробный запуск **без записи** (Dry-run) | Проверить, что всё собралось правильно |
| 7 | Сравнить «было на сервере» и «станет после правки» | Убедиться перед сохранением |
| 8 | **Откатить** к предыдущей версии из сохранённой выгрузки | Ошиблись при правке |
| 9 | Подставить длинный список из файла `.txt` | Сотни строк значений |
| 10 | Остановить длинную очередь отправки | Кнопка «Стоп» |
| 11 | **Каталог параметров** с сервера + поиск по части имени | Не ошибиться в длинном имени параметра |
| 12 | **Просмотр values** и свойств bundle на вкладке «Файл export» | Проверить содержимое перед отправкой |

### Чего скрипт **не** делает

- Не заменяет вход в СУП — вы должны быть **уже залогинены**.
- Не правит **одну строку** отдельно — всегда отправляется **весь список** значений параметра.
- Не работает с произвольной страницы интернета — только со страницы **UFS Config Manager** (см. § I.3).

## I.2. Что нужно перед началом

| Требование | Зачем | Если нет — что будет |
|------------|-------|----------------------|
| Браузер с доступом к СУП (Chrome, SberBrowser и т.п.) | Работа через веб-интерфейс | Скрипт не к чему подключиться |
| Вы **вошли** в СУП под своей учёткой | Сервер принимает только авторизованные запросы | Ошибка 401, «не удалось сохранить» |
| Открыта страница **UFS Config Manager** | Скрипт берёт адрес сервера и сессию с этой вкладки | Auto origin не сработает; запросы не пройдут |
| Файл скрипта `SUP_Config_Update.js` | Его вставляют в консоль | Панель не появится |
| **Имя параметра** (длинная строка с точками) | По нему сервер находит, что править | Не найдёт параметр |
| **Тенант** (обычно `EFSNB_GAMIFICATION`) | У каждого параметра свой «контур» в СУП | Параметр не найдётся или уйдёт не туда |

## I.3. Где запускать скрипт (важно!)

Скрипт нужно запускать **на той же вкладке**, где открыт СУП, в адресной строке которой есть путь вида:

```text
…/ufs-config-manager/…
```

**Если открыть скрипт на другой странице** (другой сайт, другой раздел портала, пустая вкладка):

- кнопка **Auto origin** напишет *«Не удалось определить ufs-config-manager»* — это **нормально**, не поломка;
- панель откроется, но **сохранить на сервер, скорее всего, не получится** — нет доступа к вашей сессии СУП.

**Вывод:** для реальной работы всегда открывайте СУП → затем консоль → затем скрипт.

## I.4. Как открыть панель скрипта (первый раз)

Пошагово:

1. Откройте **UFS Config Manager** в браузере и дождитесь загрузки (вы видите привычный интерфейс СУП).
2. Нажмите **F12** (или правой кнопкой на странице → «Просмотреть код» / Inspect).
3. Перейдите на вкладку **Console** (Консоль).
4. Откройте файл `SUP_Config_Update.js` в блокноте, выделите **весь** текст (**Ctrl+A**), скопируйте (**Ctrl+C**).
5. Кликните в поле консоли, вставьте (**Ctrl+V**), нажмите **Enter**.
6. Справа внизу (или поверх страницы) появится тёмная панель **«СУП — обновление параметров»**.

| Если не сделать | Что будет |
|-----------------|-----------|
| Не вставить **весь** файл | Ошибка в консоли, панель не появится |
| Не нажать Enter | Скрипт не запустится |
| Консоль на другой вкладке | Панель появится не там, где СУП — запросы не сработают |

Чтобы закрыть панель — кнопка **Закрыть** внизу панели. Чтобы открыть снова — повторите шаги 4–5 (перезагрузка страницы СУП не обязательна).

## I.5. Главные правила (прочитайте до правок на PROD)

### Правило 1. Отправляется весь список целиком

Когда вы нажимаете **Отправить**, на сервер уходит **полный список** всех значений параметра — не одна изменённая строка.

| Ситуация | Последствие |
|----------|-------------|
| Вы удалили из списка 400 строк и отправили только 100 | На сервере **останутся только эти 100** — остальные пропадут |
| Вы добавили одну новую строку, но забыли остальные | Старые значения **исчезнут** |

**Что делать:** перед правкой всегда **скачивайте** актуальный список с сервера или **сохраняйте JSON** на диск. Меняйте только нужные строки, **не удаляя** остальные без намерения.

### Правило 2. Dry-run — пробный режим без записи

При первом запуске галочка **«Dry-run (без POST)»** **включена**. Это хорошо.

| Dry-run | Что происходит при «Отправить» |
|---------|-------------------------------|
| **Включён** | Только проверка и запись в журнал внизу панели. **На сервер ничего не пишется.** |
| **Выключен** | Реальное сохранение на сервер (как «Сохранить» в СУП). |

| Если не сделать | Что будет |
|-----------------|-----------|
| Сразу снять Dry-run и нажать «Отправить», не проверив список | Можно случайно записать неверные или неполные данные на PROD |

**Рекомендуемый порядок:** сначала с Dry-run → смотрите журнал → убедились → снимите галочку → снова **Отправить**.

## I.6. Настройка панели после открытия (каждый раз)

Выполните **один раз** после появления панели:

| Шаг | Что нажать / выбрать | Зачем | Если пропустить |
|-----|----------------------|-------|-----------------|
| 1 | **Auto origin** | Подставит адрес сервера автоматически | Ошибка «Укажите origin…» при любой операции |
| 2 | В списке **Tenant** выбрать нужный (часто `EFSNB_GAMIFICATION`) | Параметры привязаны к тенанту | Параметр «не найден» |
| 3 | (По желанию) **Загрузить tenantCodes** | Обновит список тенантов с сервера | Можно выбрать тенант вручную, если знаете код |
| 4 | Убедиться, что **Dry-run** включён | Безопасная первая проверка | Риск записи на сервер без проверки |

Поля **Origin**, **Prefix**, **Referer** обычно трогать **не нужно** — их заполняет **Auto origin**.

## I.7. Основной сценарий: поправить один параметр

**У вас есть:** полное имя параметра, например  
`rmkib.enigma.gamification.badgesimagemapping`.

### Шаг 1. Скачать текущие значения

| Действие | Подробно |
|----------|----------|
| 1 | Нажмите вкладку **«Скачать с сервера»** |
| 2a | **Рекомендуется:** нажмите **«Загрузить список параметров»** — подтянется каталог для выбранного tenant |
| 2b | В поле **«Поиск по части имени»** начните вводить фрагмент имени (например `badges` или `gamification`) — в списке останутся только подходящие строки |
| 2c | **Кликните** нужную строку в списке (`id \| имя`) — имя подставится в поле «Имя параметра», id — на вкладку Payload |
| 2d | **Или** в поле «Имя параметра» вставьте **полное имя** вручную (как раньше) |
| 3 | Нажмите синюю кнопку **«Скачать с сервера»** |
| 4 | Подождите несколько секунд — в большом поле ниже появится текст (JSON) |

| Если не сделать | Что будет |
|-----------------|-----------|
| Опечатка в имени (при ручном вводе) | В журнале ошибка «параметр не найден» |
| Не загрузить список после смены tenant | Старый каталог неактуален — нажмите «Загрузить список» снова |
| Не тот тенант | То же или пустой ответ |
| Не нажали Auto origin | Ошибка про origin |

### Шаг 2. Сделать резервную копию (настоятельно рекомендуется)

| Действие | Подробно |
|----------|----------|
| 1 | Не закрывая панель, нажмите **«Сохранить JSON»** |
| 2 | Файл сохранится на компьютер (имя вида `parameters_20260611….json`) |

| Если не сделать | Что будет |
|-----------------|-----------|
| Ошибётесь при правке | Откат будет сложнее; кнопка «Откатить» помнит только последнюю выгрузку в этой сессии |

### Шаг 3. Внести правку

| Действие | Подробно |
|----------|----------|
| 1 | В том же большом поле (preview) найдите блок `"values": [` |
| 2 | Внутри квадратных скобок — список строк в кавычках, по одной на строку |
| 3 | Найдите нужную строку и **измените текст** (или добавьте новую строку в том же формате) |
| 4 | **Не удаляйте** остальные строки, если не хотите их убрать с сервера |

Пример фрагмента (упрощённо):

```text
"values": [
  "REWARD_00_01:BADGE_GURU",
  "REWARD_00_02:BadgeVascoDaGama",
  ...
]
```

Меняете, например, `BADGE_GURU` на `BADGE_GUR` — остальные строки **оставляете как были**.

### Шаг 4. Пробная отправка (Dry-run)

| Действие | Подробно |
|----------|----------|
| 1 | Убедитесь, что активна вкладка **«Скачать с сервера»** (она подсвечена синим) |
| 2 | Галочка **Dry-run** — **стоит** |
| 3 | Нажмите зелёную кнопку **«Отправить»** внизу |
| 4 | Смотрите **Журнал** внизу панели: должно быть `DRY-RUN`, число values, без красных `[ERR]` |

| Если не сделать | Что будет |
|-----------------|-----------|
| Переключились на другую вкладку | Отправится не то содержимое (см. § I.9) |
| Dry-run выключен | Сразу запишется на сервер — опасно без проверки |

### Шаг 5. Реальное сохранение на сервер

| Действие | Подробно |
|----------|----------|
| 1 | **Снимите** галочку **Dry-run** |
| 2 | Снова нажмите **«Отправить»** |
| 3 | В журнале: `[OK]`, в блоке **Bundle info** — новый id и дата |
| 4 | В интерфейсе СУП **обновите страницу** параметра (F5) и проверьте значение глазами |

| Если не сделать | Что будет |
|-----------------|-----------|
| Оставить Dry-run | На сервере ничего не изменится (вас могут ввести в заблуждение) |
| Не проверить в UI СУП | Не узнаете, применилось ли то, что нужно |

### Шаг 6. Если ошиблись — откат

| Действие | Подробно |
|----------|----------|
| 1 | Нажмите **«Откатить из export»** (оранжевая кнопка) |
| 2 | Подтвердите в окне браузера |
| 3 | Панель переключится на вкладку **«Файл export»** со **старым** списком (до последнего скачивания/загрузки) |
| 4 | Нажмите **«Отправить»** — Dry-run скрипт снимет сам |

| Если не сделать | Что будет |
|-----------------|-----------|
| После ошибки снова нажали «Скачать с сервера» | Старый снимок для отката **перезаписан** — откатить к «до ошибки» может быть нельзя (остался только файл с диска из шага 2) |

## I.8. Другие сценарии

### Сценарий B. Уже есть файл выгрузки на компьютере (`parameters_….json`)

| Шаг | Действие |
|-----|----------|
| 1 | **Auto origin**, выбрать **Tenant** |
| 2 | Вкладка **«Файл export»** |
| 3 | Кнопка «Выберите файл» → указать `.json` **или** вставить текст файла в поле и нажать **«Разобрать JSON»** |
| 4 | Появится список с **галочками** и колонкой **id** (или «—», если id ещё не подтянут) |
| 5 | **Кликните строку** (не только галочку) — внизу откроются два окна: **«Свойства выбранного bundle»** (имя, tenant, path, scope, description…) и **«Values»** (все строки values) |
| 6 | Оставьте галочки только у bundle, которые нужно отправить |
| 7 | (По желанию) **«Подтянуть id для отмеченных»** — заполнит id в списке до отправки |
| 8 | Dry-run → **Отправить** → проверить журнал |
| 9 | Снять Dry-run → **Отправить** |

| Если не нажать «Разобрать JSON» | Список пустой, «Нет задач для отправки» |
| Если все галочки сняты | Ничего не отправится |
| Не кликнули строку | Не увидите values в окне просмотра (первая строка выбирается автоматически) |

### Сценарий C. Несколько параметров из одного файла

Тот же порядок, что в сценарии B. В файле может быть 3, 10 или больше параметров — у каждого своя галочка. Между отправками скрипт делает паузу (**Пауза мс**, по умолчанию 500). При ошибке на одном параметре браузер спросит: **продолжить** остальные или **остановить**.

Кнопка **Стоп** — прервать очередь после текущего параметра.

### Сценарий D. Очень длинный список (сотни строк) — файл `.txt`

| Шаг | Действие |
|-----|----------|
| 1 | Подготовьте `.txt`: **одна строка = одно значение** параметра |
| 2 | Вкладка **Payload** |
| 3 | В поле имени — имя параметра → **Обновить id** |
| 4 | В большое JSON-поле — минимальная заготовка (можно после «Скачать с сервера» скопировать структуру) |
| 5 | **Выберите файл** у «valuesFile» → **Values из .txt** |
| 6 | Dry-run → **Отправить** → проверка → снять Dry-run → **Отправить** |

### Сценарий F. Выбрать параметр из каталога (без опечатки в имени)

| Шаг | Действие |
|-----|----------|
| 1 | **Auto origin**, выбрать **Tenant** |
| 2 | Вкладка **«Скачать с сервера»** |
| 3 | **«Загрузить список параметров»** — дождаться сообщения в журнале (`… N шт.`) |
| 4 | В **«Поиск по части имени»** ввести часть имени — список сузится |
| 5 | Клик по строке `id \| name` — имя и id подставятся автоматически |
| 6 | **«Скачать с сервера»** → правка → Dry-run → **Отправить** |

Ручной ввод полного имени в поле «Имя параметра» по-прежнему работает — каталог необязателен.

### Сценарий E. Только посмотреть разницу перед сохранением

| Шаг | Действие |
|-----|----------|
| 1 | Скопируйте отредактированный список на вкладку **Payload** (или соберите JSON вручную) |
| 2 | **Обновить id** по имени параметра |
| 3 | Нажмите **«Сравнить с сервером»** |
| 4 | В журнале: сколько было строк, сколько станет, сколько добавится/удалится |

Это **не сохраняет** на сервер — только показывает разницу.

## I.9. Все кнопки и элементы — простым языком

### Верх панели

| Кнопка / поле | Нажали — что произошло | Не нажали / не заполнили |
|---------------|------------------------|---------------------------|
| **Auto origin** | Заполнились адреса сервера | Любая операция с сервером выдаст ошибку |
| **Загрузить tenantCodes** | Список тенантов обновился с сервера | Можно выбрать тенант вручную, если знаете |
| **Tenant** | Выбран контур СУП для запросов | Неверный тенант → параметр не найден |

### Вкладки

| Вкладка | Для чего | «Отправить» берёт данные отсюда, если вкладка активна |
|---------|----------|------------------------------------------------------|
| **Payload** | Ручной ввод JSON (для опытных) | Из большого поля JSON |
| **Файл export** | Файл или вставленный JSON с диска | Только строки **с галочкой** |
| **Скачать с сервера** | Каталог / имя → скачать → править → отправить | Из поля preview после скачивания |

### Кнопки на вкладке «Скачать с сервера»

| Кнопка / поле | Действие |
|---------------|----------|
| **Имя параметра** | Ручной ввод полного имени **или** подстановка после клика по списку |
| **Поиск по части имени** | Фильтр загруженного каталога (по имени, id, описанию) |
| **Загрузить список параметров** | GET каталога `parameter/list` для выбранного tenant (постранично) |
| Список `id \| name` | Клик — выбрать параметр без опечатки |
| **Скачать с сервера** | Загрузить актуальные values выбранного параметра в поле ниже |
| **Сохранить JSON** | Сохранить то, что в поле, в файл на компьютер (бэкап) |

### Кнопки на вкладке «Файл export»

| Кнопка / блок | Действие |
|---------------|----------|
| Выбор файла | Открыть `.json` с диска |
| **Разобрать JSON** | Разобрать текст из поля в список bundle |
| **Подтянуть id для отмеченных** | `parameter/list` по имени для строк с галочкой |
| Список bundle | Галочка = отправить; **клик по строке** = просмотр внизу; колонка **id** |
| **Свойства выбранного bundle** | Имя, id, tenant, path, scope, description, roles… |
| **Values** | Полный список values выбранного bundle (только просмотр) |

### Кнопки на вкладке «Payload»

| Кнопка | Действие |
|--------|----------|
| **Обновить id** | Найти числовой id параметра по имени (заполнит поле parameterId) |
| **Values из .txt** | Подставить строки из выбранного `.txt` в JSON |
| **Собрать payload** | Проверить, что JSON написан без синтаксических ошибок |

### Общие кнопки внизу

| Кнопка | Действие | Опасно? |
|--------|----------|---------|
| **Собрать payload** | Проверка JSON (вкладка Payload) | Нет |
| **Сравнить с сервером** | Показать разницу списков | Нет |
| **Отправить** | Сохранить (или пробный прогон при Dry-run) | **Да**, если Dry-run выключен |
| **Откатить из export** | Подготовить возврат к последней выгрузке | Запись только после второго «Отправить» |
| **Стоп** | Остановить очередь из нескольких параметров | Нет |
| **Копировать fetch** | Для технических специалистов | Нет |
| **Закрыть** | Убрать панель | Нет |

### Галочки и поля

| Элемент | Смысл |
|---------|--------|
| **Dry-run** | Вкл. = проба без записи; Выкл. = реальное сохранение |
| **Пауза мс** | Пауза между несколькими параметрами (обычно не трогать) |
| **Bundle info** | После успешного сохранения — номер версии на сервере |
| **Журнал** | Все сообщения: что сделано, ошибки, итог |

## I.10. Таблица «что будет, если пропустить шаг»

| Пропущенный шаг | Последствие |
|-----------------|-------------|
| Запуск не на странице СУП | Auto origin не работает; сохранение не пройдёт |
| Не нажать **Auto origin** | Ошибка при любой операции |
| Не скачать актуальные values перед правкой | Можете править устаревший список и перезаписать чужие изменения |
| Не сделать **Сохранить JSON** (бэкап) | Сложный откат при ошибке |
| Удалить лишние строки из `values` | Эти значения **исчезнут** на сервере |
| Не проверить с **Dry-run** | Риск сразу записать неверные данные |
| Забыть **снять Dry-run** | Думаете, что сохранили — на сервере старое |
| Не обновить страницу в UI СУП | Не увидите результат глазами |
| После ошибки снова «Скачать с сервера» до отката | Потеряете снимок для кнопки «Откатить» |
| Неверный **Tenant** | Параметр не найден или запись не туда |
| Истекла сессия (давно не работали в СУП) | HTTP 401 — перелогиньтесь |

## I.11. Сообщения в журнале — что они значат

| Текст в журнале | Значение | Ваши действия |
|-----------------|----------|---------------|
| `Панель SUP_Config_Update готова` | Всё ок, можно работать | — |
| `Origin auto: https://…` | Адрес сервера подставлен | — |
| `DRY-RUN Очередь: 1 POST` | Проба прошла, на сервер **не писали** | Если всё верно — снять Dry-run и отправить снова |

| `Не удалось определить ufs-config-manager` | Скрипт не на странице СУП | Открыть СУП и запустить снова |

| `Id подтянуты: ok=…` | Batch resolve id на вкладке export | — |
| `Итог: ok=1 fail=0` | Очередь завершена успешно | — |
| Окно «Продолжить?» | Один параметр из пачки упал с ошибкой | Выбрать: продолжить остальные или остановить |

# Часть II. Техническая документация

## II.1. Назначение

Браузерный DevTools-скрипт для **обновления значений параметров** в UFS Config Manager (СУП) через REST API — аналог «Сохранить» в UI.

## II.2. Запуск

1. Открыть вкладку **UFS Config Manager** (авторизованная сессия).
2. DevTools → Console → вставить **полный** файл скрипта → Enter.
3. Панель `id=sup-config-update-panel`.
4. Кнопка **Auto origin** подставляет origin, API prefix и referer с текущей вкладки.

## II.3. API

Базовый prefix: `{origin}{path}/ufs-config-manager/pacman/rest/`

| Метод | Path | Назначение |
|-------|------|------------|
| GET | `tenantCodes` | Список тенантов (dropdown cfg-rn) |
| POST | `parameter/list` | Каталог tenant (без `name` в filter) или поиск по `name` → `id` |
| POST | `parameter/data/export` | Скачивание текущих values |
| POST | `parameter/bundle/list` | Активный bundle (diff, блок info) |
| POST | `parameter/value/add` | **Сохранение** (полная замена values) |

Заголовки: `cfg-rn` = tenant; `x-cfga-location` = `""`; `credentials: include`.

### Payload add

```json
{
  "parameterId": 361251,
  "bundle": {
    "path": [{ "code": "SUBSYSTEM", "value": "KKSB_ENIGMA" }],
    "values": ["REWARD_00_01:BADGE_GURU"]
  }
}
```

## II.4. Вкладки панели

| Вкладка | Описание |
|---------|----------|
| **Payload** | Ручной JSON; lookup `parameterId`; подстановка values из `.txt` |
| **Файл export** | Paste / file; EXPORT / ADD_READY / JOB; чекбоксы; колонка id; панели «Свойства» + «Values» |
| **Скачать с сервера** | Каталог `parameter/list` + поиск; ручное имя; `parameter/data/export` → preview |

## II.5. Форматы файлов

- **EXPORT** — массив записей UI export (`parameters_*.json`); `path[].name` → `path[].code`.
- **ADD_READY** — `{ meta, requests: [{ parameterId?, bundle }] }`.
- **JOB** — `{ tenant, parameters: [{ name, bundles }] }`; поле `valuesFile` — подсказка (values грузятся отдельным `.txt`).

## II.6. Безопасность

- **Dry-run** включён по умолчанию (без POST).
- **Сравнить с сервером** — diff счётчиков values с активным bundle.
- **Откатить из export** — готовит очередь из последнего export (файл или сервер).
- При ошибке в batch — диалог «продолжить / остановить».
- Предупреждение: `value/add` заменяет **весь** список values.

## II.7. Ключевые функции (код)

| Имя | Назначение |
|-----|------------|
| `detectEnvFromLocation` | Origin + API prefix с вкладки |
| `resolveParameterId` | POST parameter/list по `name` + кэш |
| `fetchAllParameters` | POST parameter/list постранично (каталог tenant) |
| `exportPathToAddPath` | Универсальный name→code в path |
| `parseImportJson` | EXPORT / ADD_READY / JOB |
| `postValueAdd` | POST value/add (или dry-run preview) |
| `fetchActiveBundle` | Активный bundle для diff и UI-блока |

## II.8. Справочник интерфейса (все поля и кнопки)

Скрипт повторяет действие **«Сохранить»** в UI СУП: отправляет `POST parameter/value/add` с **полным** списком `values`. Работает только на открытой вкладке UFS Config Manager с активной сессией (`credentials: include`).

### II.8.1. Верхняя часть панели

| Элемент | Назначение |
|---------|------------|
| **Origin** | Базовый URL стенда |
| **Auto origin** | Подставляет Origin, Prefix, Referer (нужен `/ufs-config-manager` в URL) |
| **Prefix** | `…/ufs-config-manager/pacman/rest/` |
| **Referer** | Referer для fetch |
| **Tenant (cfg-rn)** | Код тенанта |
| **Загрузить tenantCodes** | GET tenantCodes |
| **Пауза мс** | Задержка между POST (500 по умолчанию) |
| **Dry-run** | Без реальной записи на сервер |
| **⚠ полная замена values** | Предупреждение об отправке всего списка |

### II.8.2. Вкладка «Скачать с сервера»

| Элемент | Назначение |
|---------|------------|
| Имя параметра | Ручной ввод или выбор из каталога |
| Поиск по части имени | Клиентский фильтр списка (`includes` по name, id, description) |
| Загрузить список параметров | `fetchAllParameters` — pageSize 100, filter `{ tenantCodes }` |
| Список | Клик → `exportNameInput`, `paramNameInput`, `paramIdInput` |
| Скачать / Сохранить JSON | export API и локальный файл |

При смене tenant каталог сбрасывается.

### II.8.3. Вкладка «Файл export»

| Элемент | Назначение |
|---------|------------|
| Разобрать JSON | `parseImportJson` → `importRows` |
| Список | `checkbox` + id + label; клик → `showImportRowDetail` |
| Свойства / Values | Readonly textarea; meta из export + lazy `ensureRowParameterId` |
| Подтянуть id для отмеченных | Batch `resolveParameterId` для `checked` строк |

## II.11. Частые ошибки

| Симптом | Причина / решение |
|---------|-------------------|
| «Укажите origin…» | Не на вкладке СУП или не нажали **Auto origin** |
| «Не удалось определить ufs-config-manager» | Скрипт запущен не на странице СУП — ожидаемое поведение |

| HTTP 401 | Перелогиниться в СУП |
