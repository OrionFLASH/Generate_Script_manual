# Документация: скрипт загрузки профиля героев

## 1. Назначение

Скрипт предназначен для пакетной загрузки профилей сотрудников (героев) через внутренний API геймификации. Он выполняется в консоли разработчика браузера на странице соответствующего стенда, последовательно запрашивает данные по списку табельных номеров (ТН), обрабатывает ответы (в том числе опционально — фото) и сохраняет результаты в JSON-файлы на компьютер пользователя.

**Расположение скриптов:** папка `Script/` в корне проекта.

Локальные **копии** всех скриптов из `Script/`, файлов из `Docs/` и корневого `README.md` с добавлением **`.txt`** к имени файла (например, `File_DB_Load_GP.js.txt`) собираются в каталог **`POST/`**. Каталог **не коммитится** в git (см. `.gitignore`). Обновить копии: из корня репозитория выполнить `./post_txt_sync.sh`. Общее описание структуры проекта — в `README.md`.

- **Базовый скрипт** — `СКРИПТ ЗАКРУЗКА ПРОФИЛЯ ГЕРОИ (Photo - size (Стата_Фото)).js`: табельные задаются только массивом `TAB_NUMS` в коде; при вставке в консоль сбор запускается сразу.
- **Вариант с выбором источника ТН** — `..._from_file.js`: панель с кнопками «Выбрать файл .txt», **многострочное поле** + «Запустить по тексту из поля» (те же правила разбора и нормализации, что у файла, через `parseTabNumbersFromText`) и «Запустить по массиву из скрипта» (`TAB_NUMS`). Скрипт **`Profile_GP_LOAD_file.js`** использует такую же панель (в т.ч. поле ввода и выбор стенда). В JSON сохраняются все ответы как есть; при ответе API с `success: false` в **журнал работы** на панели выводится ERROR и поля `code`, `system`, `text` из объекта `error`.

## 2. Режимы работы: выбор стенда

Скрипт поддерживает два стенда. В **`Profile_GP_LOAD_file.js`** переключение — **выпадающий список** на панели (`PROFILE_UI_STAND`, по умолчанию `DEFAULT_PROFILE_STAND`). В базовом скрипте и в `..._from_file.js` без доработки по-прежнему используется константа **`STAND`** в коде.

| Стенд   | Описание |
|--------|----------|
| **ALPHA** | Промовый стенд (omega). Запросы уходят на хост вида `https://***.omega.***/...`. Запуск — со страницы этого же домена. |
| **SIGMA** | Стенд на домене вида `https://***.salesheroes.***`. Требует запуска со страницы того же домена и отправки куки сессии (`credentials: "include"`). В заголовках запроса передаются `Origin` и `Referer` с путём вида `…/profile/<табельный_номер>`. |

Для каждого стенда используется свой базовый URL API; путь к методу профиля один и тот же (маска: `…/bo/rmkib.gamification/proxy/v1/profile`). Тело запроса (payload) для обоих стендов одинаковое: объект `{ employeeNumber: "<табельный_номер>" }`.

## 3. Константы и настройки

### 3.1. Список табельных номеров

- **Переменная:** `TAB_NUMS`
- **Тип:** массив строк.
- **Назначение:** в базовом скрипте — единственный источник ТН; в `_from_file.js` и `Profile_GP_LOAD_file.js` — при выборе «Запустить по массиву из скрипта» (поле ввода и файл не используют `TAB_NUMS`).
- **Пример (маскированный):** `["00001234", "00005678"]` — подставьте реальные ТН из вашей системы.

### 3.2. Параметры прогона (`Profile_GP_LOAD_file.js`)

В коде задаются **значения по умолчанию** (префикс `DEFAULT_`); на панели их можно изменить перед запуском. Итоговые значения нормализует **`normalizeRunOptions`**, объект по умолчанию — **`getDefaultRunOptions()`**.

**Панель (UI):** ширина окна порядка **920px** (ограничение `min(920px, 100vw − отступы)`). Блок **«Параметры запуска»** свёрнут в `<details>` (по умолчанию раскрыт): поля **пауза между запросами**, **доп. попыток после первой**, **пауза перед повтором** и чекбокс **«Повтор при ошибке по ТН»** выстроены **в одну горизонтальную линию** (при узкой ширине возможна горизонтальная прокрутка строки). Строка **«Записей в батче»** и два чекбокса про фото (**скачивание отдельно**, **урезать base64**) — **в один ряд** (при нехватке места чекбоксы переносятся). У подписей, полей и чекбоксов заданы краткие подсказки при наведении (**атрибут `title`**). Три кнопки запуска (**текст в поле**, **файл .txt**, **TAB_NUMS**) — **в одну строку** сеткой из трёх колонок. Ниже — блок **«Журнал работы»** (прокручиваемая область): туда дублируются сообщения, которые скрипт выводит в консоль во время сбора (через **`profileGpPanelEcho`**). Кнопка **«Закрыть панель»** снимает панель с экрана и обнуляет приёмник журнала (**`profileGpPanelLogAppend = null`**), чтобы не оставалась ссылка на удалённый DOM; при повторном открытии панели предыдущий экземпляр удаляется с тем же сбросом журнала.

| Константа | Смысл |
|-----------|--------|
| `DEFAULT_REQUEST_DELAY_MS` | Пауза между запросами по ТН (мс). |
| `DEFAULT_ENABLE_RETRY` | Включить повтор при ошибке. |
| `DEFAULT_MAX_RETRIES` | Число **дополнительных** попыток после первой. |
| `DEFAULT_RETRY_DELAY_ON_ERROR_MS` | Пауза перед повтором (мс). |
| `DEFAULT_OUTPUT_BASE_NAME` | Базовое имя JSON (безопасные символы, обрезка в нормализации). |
| `DEFAULT_BATCH_SIZE` | Записей в одном файле до ротации. |
| `DEFAULT_ENABLE_PHOTO_DOWNLOAD` / `DEFAULT_ENABLE_PHOTO_STRIP` | Флаги фото (см. ниже). |
| `photoDownloadLinkHost` (в объекте опций) | DOM-узел панели для ссылок ручного скачивания фото; из кода обычно не задаётся (панель подставляет при запуске). |

Вызов из консоли: `runCollectProfiles(массивТН, { requestDelayMs: 10, … })` — необязательный второй аргумент частично переопределяет defaults.

### 3.3. Задержка, батч, имя файла (прочие скрипты профиля)

В базовом скрипте и в `_from_file.js` без доработки по-прежнему используются имена **`REQUEST_DELAY_MS`**, **`OUTPUT_BASE_NAME`**, **`BATCH_SIZE`** и т.д., если в файле не переименованы под модель `Profile_GP_LOAD_file.js`.

### 3.4. Флаги обработки фото

- **Скачивание фото** — при включении скрипт дополнительно скачивает `photoData` и `photoDataKpk` (base64) как `<ТН>_photoData.jpg`, `<ТН>_photoDataKpk.jpg` (программный клик по ссылке). Если файл в загрузки не попал, на панели в блоке **«Ссылки на фото»** появляются обычные ссылки — нажмите вручную (часть браузеров блокирует автоскачивание после асинхронного `fetch`).
- **Урезание из JSON** — удаление полей base64 и запись `photoDataInfo` / `photoDataKpkInfo` (`hasData`, `length`).

### 3.5. Стенд и URL

- **`Profile_GP_LOAD_file.js`:** **`DEFAULT_PROFILE_STAND`**, **`PROFILE_UI_STAND`** — значение по умолчанию и выбор на панели перед запуском сбора.
- **Прочие варианты скрипта профиля:** **STAND** — строка `"ALPHA"` или `"SIGMA"` в коде.
- **PROFILE_URLS** — объект с ключами `ALPHA` и `SIGMA`; значения — полные URL метода профиля для каждого стенда (в коде задаются реальные адреса; в документации не приводятся).
- **SIGMA_ORIGIN** — базовый origin для стенда SIGMA (хост без пути), используется для заголовков `Origin` и `Referer`.

## 4. Функции

### 4.1. Формирование тела запроса

- **Функция:** `makeRequestBody(tn)`
- **Аргумент:** `tn` — табельный номер (строка), например `"00001234"`.
- **Возвращает:** объект `{ employeeNumber: tn }` для отправки в теле POST-запроса.
- **Использование:** вызывается из `fetchProfileByTN` перед каждым запросом.

### 4.2. Вспомогательные утилиты

- **delay(ms)** — возвращает Promise, разрешающийся через заданное число миллисекунд. Используется для паузы между запросами в цикле.
- **getTimestamp()** — возвращает строку метки времени в формате `YYYYMMDD_HHMMSS` (например, `20260218_143052`). Используется в именах сохраняемых файлов.
- **getJsonSizeBytes(obj)** — возвращает размер JSON-представления объекта в байтах (через `JSON.stringify` и `TextEncoder`). Нужна для логирования размера ответов до и после обработки.

### 4.3. Работа с фото (base64)

- **decodeBase64ToBinaryString(s)** — подготовка строки к `atob`: удаление пробелов, дополнение padding до кратности 4, при ошибке — вариант **URL-safe** (`-` → `+`, `_` → `/`).
- **appendPhotoDownloadLink(hostEl, objectUrl, filename)** — добавляет на панель видимую ссылку с атрибутом `download` (ручное скачивание).
- **downloadBase64File(rawOrDataUrl, filename, linkHost)** — декодирует base64 в **Blob** и скачивает через `URL.createObjectURL` (работает и **«голый» base64** без префикса `data:image/...;base64,`). Если передан **linkHost** (контейнер на панели), дублирует тот же blob-URL ссылкой для ручного клика; отзыв `URL.revokeObjectURL` отложен дольше (чтобы ссылка оставалась рабочей). При ошибке — предупреждение в **журнал работы** (`profileGpPanelEcho("warn", …)`).

### 4.4. Обработка полей фото в ответе

- **Функция:** `findProfilePhotoContainer(data)` — ищет объект с непустыми строками `photoData` / `photoDataKpk` внутри `data.body` (в т.ч. во вложенных объектах), чтобы учесть разную структуру ответа стенда.
- **Функция:** `processPhotos(tn, data, cfg)` (`Profile_GP_LOAD_file.js`)
- **Аргумент `cfg`:** `enablePhotoDownload`, `enablePhotoStrip`, опционально **`photoDownloadLinkHost`** — DOM-узел для ссылок ручного скачивания (задаётся панелью при запуске сбора).
- **Назначение:** при необходимости скачивает фото и/или урезает base64 в JSON (см. п. 3.4).
- **Возвращает:** изменённый объект `data`.

### 4.5. Запрос профиля по табельному номеру

- **Функция:** `fetchProfileByTN(tn, cfg)` (async) в `Profile_GP_LOAD_file.js`; в других вариантах скрипта может быть одна арность.
- **Аргументы:** `tn` — табельный номер; `cfg` — нормализованные опции (фото).
- **Логика:**
  1. Формирует тело запроса через `makeRequestBody(tn)`.
  2. Выбирает URL из `PROFILE_URLS` по активному стенду: в `Profile_GP_LOAD_file.js` — `PROFILE_UI_STAND`, в остальных вариантах — `STAND` (при неверном значении используется ALPHA).
  3. Собирает заголовки: для обоих стендов — `Accept`, `Content-Type`; для SIGMA дополнительно — `Origin` и `Referer` (в Referer подставляется путь вида `…/profile/<tn>`).
  4. Для SIGMA в опции fetch добавляет `credentials: "include"` (отправка куки сессии).
  5. Выполняет POST с телом `JSON.stringify(bodyObj)`.
  6. При неуспешном HTTP (не 2xx) возвращает объект с полями `tn`, `error: true`, `status`.
  7. Парсит JSON. Если в ответе `success: false` и есть объект `error`, в **журнал работы** выводится строка с меткой ERROR и полями `code`, `system`, `text` из `error` (через **`profileGpPanelEcho`**); ответ всё равно сохраняется в общий результат. Иначе в журнал — OK и размеры. Возвращает объект с полями `tn`, `requestBody`, `processed`, `sizeBefore`, `sizeAfter`.

### 4.6. Сохранение результата в файл

- **Функция:** `saveJsonToFile(data, baseName, partIndex)`
- **Аргументы:**  
  - `data` — массив объектов (результаты по каждому ТН);  
  - `baseName` — базовое имя файла;  
  - `partIndex` — номер части (опционально); если передан, к имени добавляется `_partN`.
- **Поведение:** формирует имя файла вида `<baseName>_part<N>_<YYYYMMDD_HHMMSS>.json`, создаёт Blob с JSON, инициирует скачивание через временную ссылку и логирует количество записей и размер файла.

### 4.7. Основной цикл сбора

- **Функция:** `runCollectProfiles()` (базовый скрипт) или `runCollectProfiles(tabNums [, runOpts])`.
- **Назначение:** для каждого ТН вызывает `fetchProfileByTN` с опциями; батчирование и паузы по полям из `normalizeRunOptions(runOpts)`.
- **Старт прогона:** перед циклом, если задан `photoDownloadLinkHost`, контейнер очищается и для предыдущих ссылок вызывается `URL.revokeObjectURL` (атрибут `data-blob-url` у якорей).
- **`Profile_GP_LOAD_file.js`:** скрипт обёрнут в **IIFE** — повторная вставка в консоль без перезагрузки страницы и без ошибки повторного `const`. Панель с блоком **«Параметры запуска»**, источник ТН, три кнопки запуска в ряд, **журнал работы** и **«Закрыть панель»**; при открытии снимается предыдущая панель `id=profileGpLoadPanelRoot` с обнулением **`profileGpPanelLogAppend`**. Ход сбора в **журнал** пишется через **`profileGpPanelEcho("log"|"warn"|"error", …)`** (`log` / `warn` — только панель; `error` — панель и **`console.error`**). В **консоли** отдельно: строка при открытии панели, при старте сбора — что запущен прогон и сколько ТН, по завершении — краткая статистика (успехи / ошибки / записи). После запуска сбора панель **остаётся**; закрытие — кнопка **«Закрыть панель»** (сброс выбора файла и отвязка журнала). Сбор идёт в фоне — смотрите **журнал на панели** и загрузки браузера.

### 4.8. Лог на панели (`Profile_GP_LOAD_file.js`)

| Имя | Назначение |
|-----|------------|
| **`profileGpPanelLogAppend`** | Переменная: функция `(line: string) => void`, дописывающая строку в блок «Журнал работы», или **`null`** после закрытия панели / перед удалением старого экземпляра. |
| **`profileGpPanelEcho(level, …args)`** | `log` и `warn` — только в **журнал работы** (одна строка из аргументов после `level`). `error` — в журнал и в **`console.error`**. |

**Пример (из консоли после вставки скрипта):** прямой вызов `profileGpPanelEcho` из глобальной области недоступен — функция объявлена внутри IIFE. Для расширения логики достаточно добавлять вызовы `profileGpPanelEcho` внутри того же файла рядом с существующими.

### 4.9. Вариант _from_file.js: парсинг ТН из текста и панель выбора

- **parseTabNumbersFromText(text)** — извлекает из строки все подряд идущие цифры (разделители — любые: запятая, пробел, перенос строки и т.д.). Нормализация: меньше 8 цифр → дополнение нулями слева до 8; больше 20 цифр → последние 20 цифр. Возвращает массив строк (8–20 цифр).
- **startWithChoice()** — в `Profile_GP_LOAD_file.js`: стенд, параметры прогона (см. п. 3.2), поле ТН / файл / `TAB_NUMS`, журнал, кнопки. В `_from_file.js` — упрощённая панель без настраиваемых параметров (если не синхронизирована с файлом профиля).

## 5. Порядок использования

**Базовый скрипт (только массив в коде):**

1. Открыть страницу стенда (ALPHA или SIGMA) и авторизоваться.
2. Открыть консоль (F12 → Console), вставить скрипт из `Script/`; для `Profile_GP_LOAD_file.js` — стенд на панели, ТН из поля, файла или `TAB_NUMS`; для остальных — выставить `STAND` и источник ТН по типу скрипта.
3. Скопировать весь код в консоль и выполнить — сбор стартует сразу.

**Вариант _from_file.js (файл или массив):**

1. Открыть страницу стенда и авторизоваться.
2. Открыть консоль, вставить код из `..._from_file.js`.
3. Панель: вставить ТН в текстовое поле и «Запустить по тексту из поля», либо «Выбрать файл .txt», либо «Запустить по массиву из скрипта» (`TAB_NUMS`). Разбор поля и файла — одинаковый (`parseTabNumbersFromText`).
4. После выбора сбор выполняется автоматически; результаты скачиваются как JSON-файлы.

В файлах скриптов блоки и функции снабжены комментариями на русском языке.

## 6. Важные замечания

- Для **SIGMA** скрипт обязательно нужно запускать со страницы того же домена, чтобы отправлялись куки сессии; иначе запросы могут завершаться с ошибкой доступа (например, 403).
- Ответ API с `success: false` (например, «Запись не найдена» по ТН) сохраняется в JSON как есть; в **журнал работы** на панели (`Profile_GP_LOAD_file.js`) для такого ТН выводится строка ERROR с полями `code`, `system`, `text` из объекта `error` (через `profileGpPanelEcho`; в консоль итог смотрите по завершении сбора).
- Полные URL и реальные табельные номера в документации маскируются (примеры вида `00001234`).
- Скрипт не сохраняет данные на сервер — только запрашивает профили и сохраняет результаты в виде файлов на компьютер пользователя.
