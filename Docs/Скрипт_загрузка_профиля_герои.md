# Документация: скрипт загрузки профиля героев

## 1. Назначение

Скрипт предназначен для пакетной загрузки профилей сотрудников (героев) через внутренний API геймификации. Он выполняется в консоли разработчика браузера на странице соответствующего стенда, последовательно запрашивает данные по списку табельных номеров (ТН), обрабатывает ответы (в том числе опционально — фото) и сохраняет результаты в JSON-файлы на компьютер пользователя.

**Расположение скрипта:** папка `Script/` в корне проекта (содержимое папки не попадает в репозиторий).

---

## 2. Режимы работы: выбор стенда

Скрипт поддерживает два стенда — переключение задаётся константой `STAND`.

| Стенд   | Описание |
|--------|----------|
| **ALPHA** | Промовый стенд (omega). Запросы уходят на хост вида `https://***.omega.***/...`. Запуск — со страницы этого же домена. |
| **SIGMA** | Стенд на домене вида `https://***.salesheroes.***`. Требует запуска со страницы того же домена и отправки куки сессии (`credentials: "include"`). В заголовках запроса передаются `Origin` и `Referer` с путём вида `…/profile/<табельный_номер>`. |

Для каждого стенда используется свой базовый URL API; путь к методу профиля один и тот же (маска: `…/bo/rmkib.gamification/proxy/v1/profile`). Тело запроса (payload) для обоих стендов одинаковое: объект `{ employeeNumber: "<табельный_номер>" }`.

## 3. Константы и настройки

### 3.1. Список табельных номеров

- **Переменная:** `TAB_NUMS`
- **Тип:** массив строк.
- **Назначение:** список табельных номеров (ТН) сотрудников, для которых нужно загрузить профили.
- **Пример (маскированный):** `["00001234", "00005678", "00009999"]` — подставьте реальные ТН из вашей системы.

### 3.2. Задержка между запросами

- **Переменная:** `REQUEST_DELAY_MS`
- **Тип:** число (миллисекунды).
- **Назначение:** пауза между последовательными HTTP-запросами, чтобы снизить нагрузку на API.
- **Пример:** `2` — 2 мс между запросами.

### 3.3. Имя и размер батча для сохранения

- **OUTPUT_BASE_NAME** — базовое имя сохраняемых файлов (например, `"profiles"`). К имени добавляются номер части и метка времени.
- **BATCH_SIZE** — максимальное количество записей в одном JSON-файле. При достижении этого числа формируется новый файл (например, `profiles_part1_YYYYMMDD_HHMMSS.json`, `profiles_part2_…`).

### 3.4. Флаги обработки фото

- **ENABLE_PHOTO_DOWNLOAD** — при `true` скрипт дополнительно скачивает изображения из полей `photoData` и `photoDataKpk` (base64) как отдельные файлы. Имена файлов: `<ТН>_photoData.jpg`, `<ТН>_photoDataKpk.jpg`.
- **ENABLE_PHOTO_STRIP** — при `true` из сохранённого JSON удаляются сами поля `photoData` и `photoDataKpk`, а вместо них записываются объекты `photoDataInfo` и `photoDataKpkInfo` с признаком наличия данных и длиной (в байтах). Это сильно уменьшает размер выгружаемых файлов.

### 3.5. Стенд и URL

- **STAND** — строка `"ALPHA"` или `"SIGMA"`. Определяет, на какой базовый хост и с какими заголовками отправляются запросы.
- **PROFILE_URLS** — объект с ключами `ALPHA` и `SIGMA`; значения — полные URL метода профиля для каждого стенда (в коде задаются реальные адреса; в документации не приводятся).
- **SIGMA_ORIGIN** — базовый origin для стенда SIGMA (хост без пути), используется для заголовков `Origin` и `Referer`.

## 4. Функции

### 4.1. Формирование тела запроса

- **Функция:** `makeRequestBody(tn)`
- **Аргумент:** `tn` — табельный номер (строка), например `"00001234"`.
- **Возвращает:** объект `{ employeeNumber: tn }` для отправки в теле POST-запроса.
- **Использование:** вызывается из `fetchProfileByTN` перед каждым запросом.

### 4.2. Вспомогательные утилиты

- **delay(ms)** — возвращает Promise, разрешающийся через заданное число миллисекунд. Используется для паузы между запросами в цикле.
- **getTimestamp()** — возвращает строку метки времени в формате `YYYYMMDD_HHMMSS` (например, `20260218_143052`). Используется в именах сохраняемых файлов.
- **getJsonSizeBytes(obj)** — возвращает размер JSON-представления объекта в байтах (через `JSON.stringify` и `TextEncoder`). Нужна для логирования размера ответов до и после обработки.

### 4.3. Работа с фото (base64)

- **downloadBase64File(dataUrl, filename)** — если передан валидный data URL (строка), создаёт временную ссылку и инициирует скачивание файла с заданным именем. Вызывается из `processPhotos` при включённом `ENABLE_PHOTO_DOWNLOAD`.

### 4.4. Обработка полей фото в ответе

- **Функция:** `processPhotos(tn, data)`
- **Аргументы:** `tn` — табельный номер; `data` — объект ответа API (с полем `body` и вложенными `photoData`, `photoDataKpk`).
- **Назначение:**  
  - при `ENABLE_PHOTO_DOWNLOAD === true` — скачивает фото из `photoData` и `photoDataKpk`;  
  - при `ENABLE_PHOTO_STRIP === true` — удаляет из `data.body` сами поля с base64 и подставляет объекты `photoDataInfo` и `photoDataKpkInfo` с полями `hasData` и `length`.
- **Возвращает:** изменённый объект `data`.

### 4.5. Запрос профиля по табельному номеру

- **Функция:** `fetchProfileByTN(tn)` (async)
- **Аргумент:** `tn` — табельный номер (строка).
- **Логика:**
  1. Формирует тело запроса через `makeRequestBody(tn)`.
  2. Выбирает URL из `PROFILE_URLS[STAND]` (при неверном `STAND` используется ALPHA).
  3. Собирает заголовки: для обоих стендов — `Accept`, `Content-Type`; для SIGMA дополнительно — `Origin` и `Referer` (в Referer подставляется путь вида `…/profile/<tn>`).
  4. Для SIGMA в опции fetch добавляет `credentials: "include"` (отправка куки сессии).
  5. Выполняет POST с телом `JSON.stringify(bodyObj)`.
  6. При неуспешном HTTP (не 2xx) возвращает объект с полями `tn`, `error: true`, `status`.
  7. При успехе парсит JSON, считает размер до и после `processPhotos`, логирует в консоль и возвращает объект с полями `tn`, `requestBody`, `processed`, `sizeBefore`, `sizeAfter`.

### 4.6. Сохранение результата в файл

- **Функция:** `saveJsonToFile(data, baseName, partIndex)`
- **Аргументы:**  
  - `data` — массив объектов (результаты по каждому ТН);  
  - `baseName` — базовое имя файла;  
  - `partIndex` — номер части (опционально); если передан, к имени добавляется `_partN`.
- **Поведение:** формирует имя файла вида `<baseName>_part<N>_<YYYYMMDD_HHMMSS>.json`, создаёт Blob с JSON, инициирует скачивание через временную ссылку и логирует количество записей и размер файла.

### 4.7. Основной цикл сбора

- **Функция:** `runCollectProfiles()` (async)
- **Назначение:** последовательно для каждого элемента `TAB_NUMS` вызывает `fetchProfileByTN`, накапливает результаты в массив `batch`. Когда размер батча достигает `BATCH_SIZE`, сохраняет батч через `saveJsonToFile` и сбрасывает буфер. Между запросами выполняется `delay(REQUEST_DELAY_MS)`. В конце сохраняется остаток батча (если есть) и выводится итоговая статистика (всего ТН, успешных, ошибок, суммарный размер до/после обработки).

В конце файла скрипта вызывается `runCollectProfiles()` — при вставке в консоль сбор запускается сразу.

---

## 5. Порядок использования

1. Открыть в браузере страницу нужного стенда (ALPHA или SIGMA) и авторизоваться.
2. Открыть консоль разработчика (F12 → вкладка Console).
3. Заполнить в скрипте массив `TAB_NUMS` реальными табельными номерами (в документации используются примеры вида `00001234`, `00005678`).
4. Установить `STAND` в `"ALPHA"` или `"SIGMA"` в зависимости от открытой страницы.
5. При необходимости настроить `REQUEST_DELAY_MS`, `OUTPUT_BASE_NAME`, `BATCH_SIZE`, флаги фото.
6. Вставить весь код скрипта в консоль и выполнить. Результаты начнут скачиваться как JSON-файлы при достижении размера батча и в конце прогона.

---

## 6. Важные замечания

- Для **SIGMA** скрипт обязательно нужно запускать со страницы того же домена, чтобы отправлялись куки сессии; иначе запросы могут завершаться с ошибкой доступа (например, 403).
- Полные URL и примеры реальных табельных номеров в этой документации намеренно не приводятся или маскируются (например, `00001234`) в целях безопасности и конфиденциальности.
- Скрипт не сохраняет данные на сервер — только запрашивает профили и сохраняет их в виде файлов на компьютер пользователя, запускающего консоль.
